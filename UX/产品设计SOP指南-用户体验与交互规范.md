# UI/UE 设计指南

## 1. 概述

### 1.1 目的
本指南旨在为 UI 设计师和 UE（用户体验）设计师提供统一的设计标准和最佳实践，确保所有产品界面具有优秀的视觉呈现和用户体验，提升产品可用性、易用性和用户满意度。

### 1.2 适用范围
- UI 设计师
- UE（用户体验）设计师
- 交互设计师
- 视觉设计师
- 涉及用户界面和体验设计的所有项目

### 1.3 核心原则
- **用户优先**：所有设计决策以用户需求为出发点
- **视觉一致性**：保持设计语言和视觉风格的统一
- **界面简洁**：减少视觉噪音，突出核心内容
- **交互友好**：提供清晰的操作反馈和流畅的交互体验
- **可访问性**：确保产品对所有用户群体友好
- **数据驱动**：基于用户研究和数据分析进行设计优化

## 2. UI/UE 设计流程规范

### 2.1 需求分析阶段

#### 2.1.1 用户研究（必须）
- [ ] 明确目标用户画像
- [ ] 进行用户访谈或问卷调查（至少 3-5 个目标用户）
- [ ] 分析竞品交互模式
- [ ] 整理用户痛点和需求清单

#### 2.1.2 场景定义
- [ ] 绘制用户旅程地图（User Journey Map）
- [ ] 识别关键交互触点
- [ ] 定义使用场景和边界条件
- [ ] 明确设计优先级

### 2.2 概念设计阶段 侧后

#### 2.2.1 信息架构
- [ ] 制定信息层级结构
- [ ] 确保导航深度不超过 3 层（首页-类目-详情）
- [ ] 核心功能入口不超过 2 次点击可达
- [ ] 完成信息架构图评审

#### 2.2.2 交互原型
- [ ] 绘制低保真线框图（Wireframe）
- [ ] 定义交互流程图（Flow Chart）
- [ ] 标注关键交互说明
- [ ] 进行可用性测试（至少 3 个用户）

### 2.3 详细设计阶段

#### 2.3.1 高保真原型
- [ ] 遵循设计系统/组件库规范
- [ ] 完成所有状态设计（默认、悬停、点击、禁用、加载、错误）
- [ ] 适配多终端尺寸（手机、平板、桌面）
- [ ] 完成动效设计说明

#### 2.3.2 设计走查
- [ ] 团队内部评审
- [ ] 与开发团队进行技术可行性评审
- [ ] 产品经理确认业务逻辑
- [ ] 完成设计标注和切图

### 2.4 交付与迭代

#### 2.4.1 设计交付物
- [ ] 高保真交互原型（Figma/Sketch/Adobe XD）
- [ ] 设计规范文档
- [ ] 交互说明文档
- [ ] 设计资源包（图标、图片、字体）

#### 2.4.2 上线后跟踪
- [ ] 收集用户反馈
- [ ] 分析用户行为数据（热力图、点击率、完成率）
- [ ] 制定优化计划
- [ ] 定期迭代评审（每 2 周一次）

## 3. UI 视觉设计规范与 UE 交互设计规范

### 3.1 导航设计

#### 3.1.1 导航结构要求
- **全局导航**：保持在所有页面可见，位置固定（顶部或侧边）
- **面包屑导航**：层级超过 2 层时必须提供，显示完整路径
- **标签导航**：同级内容不超过 7 个标签，超过需折叠或分组
- **底部导航（移动端）**：核心功能 3-5 个，当前页签明确高亮

#### 3.1.2 导航反馈
- 当前位置必须有明确的视觉标识（高亮、下划线、图标变化）
- 页面跳转必须有过渡动画（建议 200-300ms）
- 导航项必须有悬停状态反馈

### 3.2 操作反馈

#### 3.2.1 即时反馈原则
- **按钮点击**：提供视觉反馈（颜色变化、阴影、缩放）
- **表单提交**：显示加载状态，防止重复提交
- **数据加载**：超过 1 秒显示加载动画，超过 3 秒显示进度条
- **操作结果**：成功/失败必须有明确提示（Toast、Modal、内联提示）

#### 3.2.2 反馈规范
| 场景 | 反馈方式 | 持续时间 | 位置 |
|------|---------|---------|------|
| 成功操作 | Toast 提示 | 2-3 秒 | 顶部居中 |
| 错误提示 | 错误提示框 | 用户手动关闭 | 相关字段下方 |
| 警告信息 | 警告横幅 | 5 秒或手动关闭 | 内容顶部 |
| 确认操作 | Modal 弹窗 | 用户确认 | 屏幕居中 |

### 3.3 表单设计

#### 3.3.1 表单布局
- 单列布局优于多列布局（移动端必须单列）
- 标签位置：顶部对齐（推荐）或左对齐
- 必填项明确标识（红色星号 * 或"必填"文字）
- 合理分组，每组不超过 7 个字段

#### 3.3.2 输入规范
- **输入框**：
  - 提供占位符（Placeholder）说明
  - 显示当前输入状态（聚焦、填充、错误）
  - 字数限制必须实时显示（如：25/100）
- **错误处理**：
  - 实时验证（失焦验证，不在输入时验证）
  - 错误信息具体明确，避免"格式错误"等模糊提示
  - 错误字段标红，并在下方显示错误原因
- **帮助信息**：
  - 复杂规则提供问号图标悬浮提示
  - 格式要求提前说明（如：密码需包含大小写字母和数字）

#### 3.3.3 提交按钮
- 主按钮颜色与次要按钮区分明显
- 按钮文案明确（"保存""提交""下一步"，避免"确定""OK"）
- 提交中状态禁用按钮，显示加载动画
- 移动端按钮高度不小于 44px（符合触控标准）

### 3.4 内容展示

#### 3.4.1 信息层级
- **标题层级**：最多 3 级（H1、H2、H3），字号递减
- **文字大小**：
  - 正文：14-16px（移动端不小于 14px）
  - 辅助信息：12-13px
  - 标题：18-24px
- **对比度**：文字与背景对比度不低于 4.5:1（WCAG AA 标准）

#### 3.4.2 列表与卡片
- **列表项**：高度一致，间距统一（8px 或 16px 倍数）
- **卡片**：圆角统一（4px/8px/12px），阴影层级不超过 3 级
- **图片**：提供占位图，加载失败显示默认图
- **空状态**：必须提供空状态插画和引导操作

#### 3.4.3 数据可视化
- 图表配色符合无障碍标准（色盲友好）
- 提供图例说明
- 支持数据点悬浮展示详细信息
- 复杂图表提供导出功能

### 3.5 交互动效

#### 3.5.1 动效时长规范
| 动效类型 | 时长 | 缓动函数 |
|---------|------|---------|
| 微交互（按钮点击） | 100-200ms | ease-out |
| 页面过渡 | 200-300ms | ease-in-out |
| 内容展开/折叠 | 250-350ms | ease |
| 复杂动画 | 400-500ms | cubic-bezier |

#### 3.5.2 动效原则
- 遵循物理规律（加速度、重力感）
- 避免过度动效造成眩晕
- 提供"减少动效"选项（无障碍需求）
- 页面跳转动画统一方向（如：前进向左滑入，后退向右滑出）

### 3.6 移动端特殊规范

#### 3.6.1 触控设计
- **点击区域**：最小 44x44px（iOS）/ 48x48dp（Android）
- **手势操作**：
  - 左滑删除需提供撤销选项
  - 下拉刷新必须有明确视觉反馈
  - 双击/长按操作需提前告知用户
- **输入优化**：
  - 调起正确键盘类型（数字、邮箱、电话）
  - 表单填写后自动聚焦下一字段

#### 3.6.2 响应式适配
- 断点设置：
  - 手机：< 768px
  - 平板：768px - 1024px
  - 桌面：> 1024px
- 移动端优先（Mobile First）设计策略
- 横竖屏切换适配

### 3.7 移动端与 PC 端设计对比

#### 3.7.1 设备特性对比

| 维度 | 移动端 | PC 端 |
|------|-------|-------|
| **屏幕尺寸** | 4-7 英寸 | 13 英寸以上 |
| **交互方式** | 触控（手指）、手势 | 鼠标、键盘、触控板 |
| **使用场景** | 碎片化时间、移动中 | 办公、深度操作 |
| **网络环境** | 4G/5G/WiFi，可能不稳定 | WiFi/有线，相对稳定 |
| **注意力** | 短暂、易分散 | 长时间、专注 |
| **操作精度** | 低（手指约 44px）| 高（鼠标约 1px） |
| **屏幕方向** | 竖屏为主，可横屏 | 横屏固定 |

#### 3.7.2 布局策略对比

**移动端布局原则**：
- 单列布局，垂直滚动
- 卡片式设计，模块化信息
- 底部操作区（拇指易触达）
- 次要信息默认折叠
- 全屏模式，充分利用空间

**PC 端布局原则**：
- 多列布局，左右分栏
- 侧边栏导航常驻
- 顶部操作栏集中
- 一屏展示更多内容
- 固定布局，两侧留白

#### 3.7.3 导航模式对比

| 导航类型 | 移动端方案 | PC 端方案 |
|---------|-----------|----------|
| 主导航 | 底部标签栏（3-5个）或汉堡菜单 | 顶部横向导航或左侧垂直导航 |
| 二级导航 | 下拉展开或独立页面 | 悬停展开子菜单 |
| 面包屑 | 简化显示（仅上一级 + 当前） | 完整路径显示 |
| 返回操作 | 左上角返回按钮 + 手势滑动 | 浏览器返回按钮 |

**案例：后台管理系统导航**

✅ **正确做法**

移动端：
- 底部固定 Tab：首页、订单、数据、消息、我的
- 顶部：返回 + 页面标题 + 更多（三点菜单）
- 二级导航通过顶部下拉选择器切换

PC 端：
- 左侧垂直导航（可折叠），显示所有模块和子菜单
- 顶部：面包屑导航 + 用户信息 + 通知
- 鼠标悬停子菜单自动展开

❌ **错误做法**

移动端错误：
- 使用汉堡菜单隐藏所有导航（用户需额外点击）
- 底部 Tab 超过 5 个（操作区拥挤）
- 面包屑显示完整路径（占用过多空间）

PC 端错误：
- 使用底部标签栏（不符合桌面操作习惯）
- 导航层级过深（4 层以上）
- 没有面包屑导航（用户迷失）

#### 3.7.4 交互方式对比

| 交互场景 | 移动端 | PC 端 |
|---------|-------|-------|
| 选择操作 | 点击（最小 44px）| 点击（可更小，20px+）|
| 批量操作 | 长按进入多选模式 | Shift/Ctrl + 点击 |
| 删除操作 | 左滑显示删除按钮 | 悬停显示或右键菜单 |
| 详情查看 | 点击进入新页面 | 悬停预览或侧边抽屉 |
| 刷新内容 | 下拉刷新 | 刷新按钮或 F5 键 |
| 缩放内容 | 双指捏合 | Ctrl + 滚轮 |

**案例：邮件列表操作**

✅ **正确做法**

移动端：
```
左滑邮件 → [删除] [归档] 按钮出现
长按邮件 → 进入多选模式 → 底部操作栏显示
点击邮件 → 全屏打开邮件详情
下拉 → 刷新邮件列表
```

PC 端：
```
鼠标悬停 → 显示 [删除] [归档] [星标] 图标
Ctrl + 点击 → 多选邮件
点击邮件 → 右侧预览窗格显示（不跳转）
双击邮件 → 新窗口打开
```

❌ **错误做法**

移动端错误：
- 删除操作没有左滑快捷方式，必须进入详情页操作
- 批量删除需要逐个进入选择（无长按多选）

PC 端错误：
- 必须右键菜单才能删除（悬停不显示操作）
- 点击邮件就跳转新页面（无预览模式）

#### 3.7.5 表单设计对比

**移动端表单设计**：
- 单列布局（必须）
- 输入框高度 44-48px
- 标签在输入框上方
- 字段间距 20-24px
- 调起对应键盘类型
- 每屏 3-5 个字段

**PC 端表单设计**：
- 可多列布局（相关字段横向排列）
- 输入框高度 36-40px
- 标签可左对齐或顶部对齐
- 字段间距 12-16px
- 支持 Tab 键快速切换
- 可一屏显示更多字段

#### 3.7.6 内容密度对比

### 3.8 一次性操作 vs 频繁操作设计策略

#### 3.8.1 操作频率分类

| 操作类型 | 定义 | 频率 | 设计优先级 |
|---------|------|------|-----------|
| **频繁操作** | 用户每天/每小时执行多次 | 10+ 次/天 | 快捷、简化 |
| **常规操作** | 用户定期执行 | 2-10 次/天 | 平衡效率与安全 |
| **低频操作** | 用户偶尔执行 | < 1 次/天 | 清晰引导 |
| **一次性操作** | 用户可能只执行一次 | 首次使用 | 详细说明、强确认 |

#### 3.8.2 核心设计差异对比

| 设计维度 | 频繁操作 | 一次性/低频操作 |
|---------|---------|----------------|
| **操作步骤** | 1-2 步，极简流程 | 3-5 步，分步引导 |
| **确认机制** | 轻量提示或无确认 | 二次确认 + 风险提示 |
| **快捷方式** | 必须提供（快捷键、手势） | 不需要 |
| **界面入口** | 一级菜单、常驻显示 | 二级/三级菜单、折叠隐藏 |
| **撤销机制** | 必须支持撤销 | 操作前充分提示 |
| **批量操作** | 支持批量处理 | 单个操作即可 |
| **默认值** | 记住上次选择 | 使用安全默认值 |
| **操作反馈** | 轻量 Toast 提示 | Modal 确认对话框 |

#### 3.8.3 操作步骤简化策略

##### 频繁操作简化原则
1. **减少点击次数**：从 3 步缩减到 1-2 步
2. **提供快捷入口**：悬浮按钮、快捷键、右键菜单
3. **支持批量操作**：一次处理多个项目
4. **智能预填充**：记住上次输入，减少重复
5. **即时反馈**：无需跳转页面，原地操作

##### 一次性操作完整引导
1. **分步向导**：复杂流程拆分为多个小步骤
2. **进度提示**：显示当前步骤和剩余步骤
3. **帮助说明**：每步提供详细说明和示例
4. **保存草稿**：支持中断和恢复
5. **最终确认**：操作前预览和二次确认

#### 3.8.4 案例对比：删除操作

**场景 A：删除聊天消息（频繁操作）**

✅ **正确做法**

移动端：
```
左滑消息 → [删除] 按钮 → 轻量提示"已删除" + [撤销 3秒]
```

PC 端：
```
悬停消息 → 显示[删除]图标 → 点击删除 → Toast提示"已删除，撤销"
快捷键：选中消息 + Delete键
```

设计要点：
- ✅ 1 步操作（左滑/悬停点击）
- ✅ 无弹窗确认（不打断流程）
- ✅ 提供撤销机制（3-5 秒内）
- ✅ 支持批量删除（长按多选）

**场景 B：删除账号（一次性操作）**

✅ **正确做法**

```
设置 → 账号与安全 → 注销账号（红色警告文字）
  ↓
[第1步] 风险提示页面：
  "注销后将：
   • 永久删除所有数据
   • 无法恢复聊天记录
   • 注销后 30 天内禁止重新注册"
  [我再想想]  [继续注销]
  ↓
[第2步] 验证身份：
  输入手机验证码
  ↓
[第3步] 最终确认：
  "请输入'确认注销'以继续"
  [_____________]
  [取消]  [确认注销]
```

设计要点：
- ✅ 3 步流程，每步都有明确目的
- ✅ 风险提示前置（列出所有后果）
- ✅ 验证身份（防止误操作）
- ✅ 文字输入确认（最高安全级别）
- ✅ 无撤销机制（操作不可逆）

❌ **错误做法**

频繁删除消息却需要：
- ❌ 点击删除 → 弹窗"确定删除吗？" → 确定 → 成功提示（步骤过多）
- ❌ 无撤销选项（误删无法恢复）
- ❌ 无批量删除（需逐条操作）

删除账号却太简单：
- ❌ 一键注销，无任何确认（高风险）
- ❌ 删除后无冷静期（无法反悔）

#### 3.8.5 案例对比：发送操作

**场景 A：发送即时消息（频繁操作）**

✅ **正确做法**

```
[输入框]  [发送]  ← 一步完成
```

优化方案：
- ✅ Enter 键直接发送（PC 端）
- ✅ 发送后立即清空输入框
- ✅ 支持长按撤回（2 分钟内）
- ✅ 草稿自动保存（切换聊天时）

**场景 B：发送项目邮件给 50 个客户（低频操作）**

✅ **正确做法**

```
[第1步] 选择收件人：
  [搜索客户] + [批量导入]
  已选：50 人 [查看列表]
  ↓
[第2步] 编写内容：
  主题：[___________]
  正文：[___________]
  附件：[选择文件]
  [保存草稿]  [预览]  [下一步]
  ↓
[第3步] 预览确认：
  收件人：张三、李四...（展开全部）
  主题：XX项目通知
  正文预览：...
  发送时间：[立即发送 ▼]  或  [定时发送]

  ⚠️ 此邮件将发送给 50 人，发送后无法撤回

  [返回修改]  [确认发送]
```

设计要点：
- ✅ 分步流程（选人 → 编辑 → 确认）
- ✅ 预览功能（检查内容）
- ✅ 风险提示（群发提醒）
- ✅ 支持保存草稿
- ✅ 显示收件人数量

❌ **错误做法**

即时消息却太复杂：
- ❌ 发送前弹窗："确定发送吗？"
- ❌ 需要两步：点击发送 → 确认发送
- ❌ 无快捷键支持

群发邮件却太简单：
- ❌ 直接一键发送给所有人
- ❌ 无预览功能
- ❌ 无发送前确认

#### 3.8.6 快捷方式设计规范

##### 频繁操作必须提供的快捷方式

**移动端快捷方式**：
- 左滑/右滑手势（删除、归档）
- 长按快捷菜单（复制、分享、删除）
- 悬浮按钮（发布内容、添加项目）
- 底部固定操作栏（频繁使用的功能）

**PC 端快捷方式**：
- 键盘快捷键（Ctrl+S 保存、Ctrl+Z 撤销）
- 鼠标悬停快捷操作（编辑、删除、分享）
- 右键菜单（常用操作集合）
- 工具栏常驻（一键触达）

**案例：邮箱应用快捷键**

✅ **正确做法**

```
频繁操作快捷键：
E - 归档邮件
R - 回复
A - 全部回复
F - 转发
Delete - 删除
/ - 搜索
C - 写新邮件

低频操作需要菜单：
标记为垃圾邮件 → 右键菜单或更多选项
创建筛选规则 → 设置菜单
导出邮件 → 设置菜单
```

#### 3.8.7 批量操作设计

##### 频繁批量操作场景

✅ **正确做法**

```
文件管理器批量操作：

1. 进入批量模式：
   长按/Ctrl+点击第一个文件 → 进入多选模式

2. 选择项目：
   [全选] [反选] [已选 15 个]

3. 批量操作栏：
   ┌─────────────────────────────────┐
   │ [移动] [复制] [删除] [更多▼]   │
   └─────────────────────────────────┘

4. 执行操作：
   点击[删除] → Toast提示"已删除15个文件，撤销"
```

设计要点：
- ✅ 明确的多选状态提示（已选数量）
- ✅ 底部固定操作栏（始终可见）
- ✅ 全选/反选快捷功能
- ✅ 批量操作后统一反馈
- ✅ 支持批量撤销

❌ **错误做法**

- ❌ 批量选择后需要逐个确认删除
- ❌ 无法显示已选数量
- ❌ 批量操作入口隐藏在菜单中
- ❌ 删除 100 个文件弹出 100 次确认

#### 3.8.8 智能默认值与操作记忆

##### 频繁操作的智能优化

✅ **正确做法**

**场景：每天创建任务**

```
创建任务表单：

分类：[工作 ▼]  ← 记住上次选择（工作）
优先级：[中 ▼]   ← 记住上次选择（中）
负责人：[张三 ▼] ← 记住上次选择（张三）
截止日期：[明天]  ← 智能默认（明天）

[快速创建]  ← 使用默认值一键创建
[详细设置]  ← 修改参数后创建
```

优化点：
- ✅ 记住用户习惯（上次选择）
- ✅ 智能推荐（基于历史数据）
- ✅ 提供快速创建（一键使用默认值）
- ✅ 保留详细设置入口

**场景：首次设置账号（一次性操作）**

```
账号设置：

用户名：[_______]  ← 无默认值（必须输入）
性别：[请选择 ▼]   ← 无默认值（避免假设）
生日：[____/__/__] ← 无默认值
地区：[请选择 ▼]   ← 无默认值
```

设计要点：
- ✅ 不使用预设默认值（避免用户未注意就提交）
- ✅ 所有选项需用户主动选择
- ✅ 提供明确的占位符提示

#### 3.8.9 确认机制分级标准

| 操作风险等级 | 操作特征 | 确认方式 | 示例 |
|------------|---------|---------|------|
| **低风险（频繁）** | 可撤销、影响小 | 无需确认 + 撤销选项 | 删除单条消息、标记已读 |
| **中风险（常规）** | 可撤销、有一定影响 | 轻量确认（Toast/对话框） | 删除草稿、移出分组 |
| **高风险（低频）** | 不可撤销、影响重要数据 | 二次确认 + 说明后果 | 删除项目、清空数据 |
| **极高风险（一次性）** | 不可逆、影响账号/财产 | 多重验证 + 文字输入确认 | 注销账号、转账 |

**案例：确认机制分级**

✅ **正确做法**

低风险（删除单张照片）：
```
长按照片 → [删除] → Toast"已删除，撤销"
```

中风险（删除相册）：
```
点击[删除相册] →
  "确定删除'旅行'相册吗？包含 50 张照片"
  [取消] [删除]
```

高风险（清空回收站）：
```
点击[清空回收站] →
  "确定清空回收站吗？
   • 将永久删除 200 个项目
   • 此操作无法撤销
   • 删除后无法恢复"
  [取消] [永久删除]
```

极高风险（注销账号）：
```
[第1步] 风险告知（列出所有后果）
[第2步] 验证手机号/邮箱
[第3步] 输入"确认注销"文字
[第4步] 设置 7-30 天冷静期
```

#### 3.8.10 设计决策流程图

```
用户操作 → 判断操作频率
              ↓
        是否频繁操作（>10次/天）？
              ↓
    是 ←─────┴─────→ 否
    ↓                ↓
简化流程：        判断操作风险：
• 1-2步完成       ↓

```

## 4. 无障碍设计规范（Accessibility）

### 4.1 视觉无障碍
- [ ] 所有交互元素支持键盘操作（Tab 切换、Enter 确认）
- [ ] 焦点状态清晰可见（边框高亮）
- [ ] 颜色信息不作为唯一区分手段（配合图标、文字）
- [ ] 支持文字缩放（200% 不破坏布局）

### 4.2 语义化标注
- [ ] 图片提供 alt 文本描述
- [ ] 表单控件关联 label 标签
- [ ] 按钮文案描述准确（避免"点击这里"）
- [ ] 使用 ARIA 标签增强语义（role、aria-label）

### 4.3 多感官反馈
- [ ] 重要操作提供声音反馈选项
- [ ] 错误提示配合图标和颜色
- [ ] 支持屏幕阅读器（VoiceOver、TalkBack）

## 5. UI/UE 设计评审标准

### 5.1 评审检查清单

#### 用户体验维度
- [ ] 是否解决了用户核心问题？
- [ ] 操作流程是否最简化？（能否减少步骤）
- [ ] 是否符合用户心智模型？（与竞品/行业习惯对比）
- [ ] 异常情况是否处理完善？（网络错误、空状态、权限不足）

#### 交互规范维度
- [ ] 是否遵循设计系统/组件库？
- [ ] 交互反馈是否及时明确？
- [ ] 文案是否清晰准确？
- [ ] 是否支持多终端适配？

#### 技术可行性维度
- [ ] 是否与开发确认技术方案？
- [ ] 性能影响是否可控？（动效、大图）
- [ ] 是否考虑数据加载和容错？

### 5.2 评审流程
1. **设计师自查**：完成检查清单
2. **团队评审**：组织 30 分钟评审会议
3. **记录反馈**：整理优化点并排优先级
4. **迭代修改**：完成修改后再次走查
5. **最终确认**：产品经理和开发负责人签字确认
